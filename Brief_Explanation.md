## 1. System Working Flow
Our personalized landing page engine employs a robust multi-model pipeline via Python, abstracting complexities behind a sleek UI:
1. **Ad Ingestion & Processing**: The system takes an uploaded ad creative (Image) and utilizes OCR (Optical Character Recognition via OCR.space) to extract the primary messaging, tone, and specific offer/language directly from the image without needing heavy local vision models.
2. **Target Web Scraping**: It immediately pings the original landing page URL, intercepting its layout. We parse the Raw HTML using `BeautifulSoup`, building a tree mapping primarily conversion-heavy nodes like Headers (`h1`, `h2`), Subheaders (`h3`), body text (`p`), and interactive elements (`a` CTA tags).
3. **AI Enhancement**: Batches of text from the live site, along with the OCR'd ad copy, are bundled into a prompt engineered for CRO. Using Groq's high-speed inference (LLaMA 3.3 70B Versatile), the text elements are recompiled to specifically align the landing page text with the Ad Creative's vibe and offer, preserving tag hierarchies.
4. **Site Reassembly & Rendering**: The output JSON objects map back directly to the DOM nodes. The original HTML is intelligently overwritten in real-time, modifying only the content—not the underlying architecture. The resulting new HTML snippet is embedded as an interactive sandbox (`iframe`) directly in the UI.

## 2. Key Components & Agent Design
Our "Agent" strictly operates as a parallel NLP parser rather than a roaming crawler:
- **Streamlit Engine**: Provides instant UI, file handling, and async request orchestration. 
- **OCR Pre-Processor**: Standardizes image inputs (regardless of graphic style) into parseable string literals, bridging the gap between design tokens and language models rapidly.
- **Cheerio/BS4 Router (The Architect)**: Creates the deterministic blueprint of the site. Instead of risking AI structure manipulation, it anchors text into known DOM IDs/classes.
- **Conversion Agent (Groq / LLaMA 3.3)**: A stateless inference layer instructed strictly with a rigid JSON schema mandate. It enforces CRO principles (A/B best practices, objection handling, actionable CTAs).

## 3. Handling Edge Cases

### Random Changes
Since web pages change structurally, our strategy isolates text. By pulling only distinct structural elements (`h1`-`h3`, `p`, `a`), the AI relies entirely on semantic significance and tag grouping, ignoring volatile DOM classes (like `Tailwind` utility classes) which are left untouched.

### Broken UI
Because the system modifies the DOM in-place natively, **no CSS or HTML structure is generated by the AI**. The LLM explicitly operates inside a sandbox constrained to pure text manipulation. Injecting raw `<base href="...">` anchors relative assets implicitly so the page doesn't look "naked", avoiding broken UI styling.

### Hallucinations
A classic risk of prompting an LLM is getting irrelevant content. Our prompt strictly limits context to: `Original Landing Page Node` + `Ad Creative Literal String`. The AI outputs a pure JSON object validating 1:1 against the original ID mappings. 

### Inconsistent Outputs
If the model breaks schema (outputs markdown or trailing text), our resilient parser scrubs LLM padding elements `regex` before JSON validation. If that fails, it falls back to the original text node ensuring the site never crashes. The temperature is forced low (0.7) to restrict output variance and prioritize logical CRO over exaggerated creativity.
